Command Trigger: `/tiq:workflow:generate-context`

## Workflow Intent

Produce `.tiqora/context/global-context.md` — a comprehensive, AI-optimized context document that
combines the Technical Architect's view of the codebase with the Product Manager's view of the
current product state. This document is consumed by vibe coding agents to maximize their
performance without requiring them to re-discover the project on every session.

---

## Setup

1. Load project config from `.tiqora.yaml` (required) and `~/.tiqora/config.yaml` (optional base).
2. Merge effective config. Apply defaults if missing: `communication_language=Français`, `document_language=English`.
3. Create output directory if missing: `.tiqora/context/`.
4. Communicate progress in `communication_language`. Write all output documents in `document_language`.

---

## Phase 1 — Architecture Analysis (Architect Lens)

> Activate the Architect persona. You are Tiq Architect, Technical Architecture Lead. Produce a
> concise, structured architecture summary optimized for AI agent consumption.

### Steps

**1.1 — Entry Points**
Identify all application entry points:
- Look for: `main.ts`, `index.ts`, `app.ts`, `server.ts`, `main.py`, `app.py`, `main.go`, `main.rs`
- Note the runtime (Node.js, Python, Go, Rust, JVM, etc.)

**1.2 — Tech Stack**
Read dependency manifests (`package.json`, `requirements.txt`, `Cargo.toml`, `go.mod`, `pom.xml`).
For each major dependency, note its role (framework, ORM, test runner, build tool, etc.).

**1.3 — Directory Structure**
Map the top-level directory structure. For each major directory, describe its role in one sentence.
Go one level deeper for directories that contain core domain logic.

**1.4 — Architectural Patterns**
Identify the dominant architectural patterns in use:
- Monolith vs microservices vs monorepo
- MVC / hexagonal / clean / layered
- State management approach (Redux, Zustand, signals, etc.)
- API style (REST, GraphQL, gRPC, tRPC, etc.)
- Testing strategy (unit / integration / e2e, test frameworks)

**1.5 — Key Interfaces & Contracts**
Identify the critical shared types, interfaces, and API contracts between modules.
Cite exact file paths.

**1.6 — Tech Debt Flags**
Flag technical debt with severity:
- `[CRITICAL]` — blocks features or creates security risk
- `[MODERATE]` — degrades developer velocity
- `[MINOR]` — cosmetic or low-impact

**Output**: Write findings to `.tiqora/context/architecture.md`.

## Phase 2 — PM Context (PM Lens)

> Activate the PM persona. You are Tiq PM, Product Manager. Summarize the current product state
> for AI agent consumption.

### Steps

**2.1 — Project Purpose**
In 2–3 sentences: what does this project do, who are the primary users, what phase is it in?
Sources: README, `.tiqora.yaml` (name, pm_tool, projectKey), any PRODUCT.md or docs/ files.

**2.2 — Current Sprint**
Identify the active sprint:
- Sprint name/number, goal, end date if available
- Source: `.tiqora/sprints/`, Jira sync data in `.tiqora/context/`, or README

**2.3 — Active & High-Priority Tickets**
List tickets with status `in_progress` or `todo` + priority `high`:
- Format: `[ID] Title — brief acceptance criteria summary`
- Source: `.tiqora/` local ticket files, or fallback to `.tiqora.yaml` projectKey

**2.4 — Epics Overview**
Summarize active epics and their business context (1 sentence per epic).

**2.5 — Blockers**
Surface any tickets marked as blocked, with their dependency chain.

**Output**: Write findings to `.tiqora/context/pm-context.md`.

## Phase 3 — Synthesis (Both Lenses)

> With both `architecture.md` and `pm-context.md` in full context, produce the unified document.
> Load both files before writing. The synthesis must reflect BOTH perspectives, not just one.

### Output: `.tiqora/context/global-context.md`

Use exactly this structure:

```markdown
# [Project Name] — Global Context
> Generated by Tiqora on {date}. Re-run `/tiq:workflow:generate-context` to refresh.

## Project Overview
[2–3 sentences: what it does, who uses it, current development phase]

## Tech Stack
[Bullet list: language + version, primary framework, key libraries, infra/deployment]

## Architecture
### Entry Points
[File paths + one-line descriptions]

### Directory Map
[Top-level dirs with role descriptions]

### Key Patterns
[Dominant patterns: architecture style, state management, API style, test strategy]

### Critical Interfaces
[Key shared types / API contracts with file references]

## Current Sprint
**Goal**: [Sprint goal]
**Active tickets**:
- [ID] [Title] — [AC summary]
- ...

## Business Context
### Epics
- [Epic name]: [1-sentence business context]

### Blockers
- [Ticket ID] blocked by [dependency]

## Constraints & Tech Debt
- [CRITICAL/MODERATE/MINOR] [Description] — [file reference]
```

## Phase 4 — Distribution (Optional)

After writing `global-context.md`, ask the user:

> "Context generated at `.tiqora/context/global-context.md`.
> Would you like me to propagate it to agent configuration files?
> - [A] Write to `CLAUDE.md` (Claude Code)
> - [B] Write to `.cursorrules` (Cursor)
> - [C] Write to `llms.txt` (multi-agent standard)
> - [D] All of the above
> - [N] No, keep it in `.tiqora/context/` only"

If the user selects A, B, C, or D:
- Prepend a clear section header: `## Tiqora Project Context`
- Append the content of `global-context.md` under that header
- Do not overwrite existing content — append or insert in a clearly delimited block
- Report the files written

## Success Criteria

- `.tiqora/context/architecture.md` created and non-empty
- `.tiqora/context/pm-context.md` created and non-empty
- `.tiqora/context/global-context.md` created with all sections populated
- Every technical claim in architecture.md cites a file path
- Document is readable by an AI agent with zero prior project knowledge
