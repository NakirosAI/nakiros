## ANALYSIS REQUIREMENTS

### 1 — Product role of this repo

Identify what product responsibility this repo seems to own today.
- Ground it in README files, routes, screens, handlers, controllers, services, domain modules, and integration docs.
- Distinguish confirmed evidence from inference inside the wording.

### 2 — Primary actors visible in this repo

List the users, operators, systems, or partner roles clearly implied by this repo.
For each actor:
- actor
- what they seem to do here
- evidence path

### 3 — Capabilities implemented in this repo

List the main product capabilities this repo appears to support.
For each capability:
- name
- short summary
- evidence paths

### 4 — Business workflows visible in this repo

Identify the main user or system workflows represented in this repo.
For each workflow:
- name
- summary
- evidence paths
- confidence

### 5 — Domain glossary

Extract the important domain terms this repo uses.
For each:
- term
- meaning
- evidence path

### 6 — Product-facing dependencies

Identify how this repo depends on or feeds other repos from a product point of view.
Examples:
- UI depends on backend contract
- backend emits events consumed elsewhere
- admin flows trigger actions in another service

For each dependency:
- repo
- kind (`upstream`, `downstream`, `shared-domain`, `async-flow`)
- reason
- evidence paths

### 7 — Legacy / migration areas

Identify transitions visible in this repo:
- old model still present
- target model
- coexistence mechanism
- source of truth when inferable
- evidence paths

### 8 — Functional risks

List product-facing or operator-facing risks:
- partial feature parity
- unclear ownership
- migration ambiguity
- weak observability for a key workflow
- inconsistent UX or contract behavior

Each item:
- severity
- description
- evidence path

### 9 — Confidence

State:
- overall confidence
- reasoning
- unknowns
