## ANALYSIS REQUIREMENTS

### 1 — Workspace product purpose

Explain what the overall product does today by combining the repo-scoped findings.
Be explicit when the workspace purpose is confirmed vs inferred from multiple repos.

### 2 — Primary users / actors

Aggregate the actors found across repos.
- deduplicate equivalent roles
- keep repo-specific nuance when needed
- include evidence paths

### 3 — Core capabilities

Build a workspace capability map.
For each capability:
- name
- short summary
- repos involved
- evidence paths

### 4 — Main business workflows

Build the end-to-end workflows that exist across the workspace.
For each workflow:
- name
- summary
- repos involved
- evidence paths
- confidence

### 5 — Repo responsibilities

Explain how each repo contributes to the product:
- repo
- product_role
- supports

### 6 — Legacy / migration areas

Merge migration findings across repos:
- legacy model
- target model
- coexistence pattern
- source of truth
- evidence paths

### 7 — Functional risks

Aggregate the main product-facing and cross-repo risks:
- unclear ownership
- migration inconsistencies
- workflow gaps
- contract ambiguity
- operator confusion risk

Each item:
- severity
- description
- evidence path

### 8 — Inter-repo system view

For multi-repo workspaces, build:
- dependency graph
- source of truth matrix
- shared contracts
- end-to-end flows
- change routing
- cross-repo risks

Use repo PM findings for product flows and repo Architect findings for technical boundaries.

### 9 — Confidence

State:
- overall confidence
- reasoning
- unknowns that still block a strong workspace view
